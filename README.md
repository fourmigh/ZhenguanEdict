# ZhenguanEdict

**历代官制 × Loop Engineering × 多 Agent 协作框架**

将中国历代治理智慧的演进，映射为多智能体协作的拓扑模式。从炎黄部落的二人分工到清代军机处的极速闭环——每个朝代代表一种 Agent 协作范式。

---

## 核心理念

单个 LLM Agent 不可靠，但**制度化的协作系统**可以可靠。

中国古代官制用了一千多年解决一个核心问题：如何让一群各有私心的人协同产出稳定的治理结果。答案是——**分权制衡、专职审核、完整留痕**。这套思路放在多 Agent 系统中，比 CrewAI、AutoGen 等现代框架更早洞见了"制度性审核"的必要性。

ZhenguanEdict 不是模仿 Edict 做一个朝代，而是把**整个官制进化史**做成一幅可交互的拓扑图谱。

| 夏   | 3       | 王权 + 神权 + 工权      | 占卜即 QA                 | 最早的审查节点 |
| 商   | 4       | 神权主导，贞人代天言事  | Maker-Checker-Logger      | 审计日志起源   |
| 周   | 5+      | 封建分权，诸侯自治      | Worktree 并行沙箱         | 隔离执行       |
| 秦   | 6       | 法治集权，绝对路由      | 确定性流水线              | 硬编码路由     |

| 隋唐 | 12      | 三省六部，封驳制度      | 完整审核闭环              | 代际性审核     |

| 明   | 10+     | 内阁预审 + 司礼监批红   | 快捷通道 + 隐藏审查       | 双轨制         |

## 架构示意

```
User / Dashboard (单文件 HTML + Vanilla JS)
        │  朝代切换 · 创建任务 · 查看奏折
        ▼
┌─────────────────────────────┐
│       Dynasty Engine         │
│  拓扑加载 · 路由调度 · 状态机 │
│  Agent 管理 · 奏折归档       │
└──────────┬──────────────────┘
           │ HTTP (OpenAI 兼容)
┌──────────▼──────────────────┐
│    llama.cpp (Router Mode)   │
│  规划型 · 编码型 · 审查型    │
│  文档型 · 轻量型              │
└─────────────────────────────┘
```
